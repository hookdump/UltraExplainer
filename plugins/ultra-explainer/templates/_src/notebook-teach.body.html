Data structures · probabilistic · TEACH mode

A Bloom filter can be certain that it has never seen a value — while storing none of the values it's seen. How can a no be proof, but a yes only a hunch?

We'll build one by hand, watch it answer two queries, then you'll predict the tricky case before it's revealed. No prior knowledge beyond "a hash function turns a word into a number."

Hold your current picture for a second. You probably imagine a set as a list of the things you put in it — to answer "is x in here?" it scans for x. A Bloom filter does something stranger: it throws the values away entirely and keeps only a row of bits. That's why it's tiny — and why it can only ever say "maybe."

Why two hash functions? One bit per word collides far too often. Real filters use k hashes over m bits; here k=2, m=12 to keep it legible. Step through with ←/→ once the player has focus.
The asymmetry to watch. add only ever turns bits on — never off. So a bit still sitting at 0 proves nothing was ever hashed there. That's the whole trick, and it lands on the next panel.
The one idea

A single 0 is a proof of absence.

Because add only ever sets bits to 1, a position that is still 0 can only mean nothing was ever hashed there. So if any of a value's bits is 0, that value was definitely never added. The reverse isn't true: all-ones can happen by accident.

Now you predict

Same filter as the end of the walkthrough — bits 2, 4, and 7 are set. We ask maybe('cow'), where h1('cow')=2 and h2('cow')=4. Both of those bits are currently 1.

Commit to an answer first: what does the filter return, and is it telling the truth?

Reveal the answer

It returns maybe present — both bits are set. But cow was never added. Bits 2 and 4 were turned on by cat and dog; cow just happened to hash onto the same two. This is a false positive.

That's the deal a Bloom filter makes: it can false-positive, but it can never false-negative. A no is a guarantee. A yes is a strong hint you still have to verify.

Honest analogy. It's the guest list at the door — but only half of it. "Your name isn't on the list" is final; you're not getting in. "I see your name" still sends you to have your ID checked, because two guests can share a name. The filter is the quick list scan; the real database lookup is the ID check. Where it breaks: a real guest list can also give a false no (someone forgot to add you) — a Bloom filter never can.
Exit ticket

You put a Bloom filter in front of a slow database so you can skip lookups for keys that don't exist. A query returns "maybe."

What do you do next — and what did the filter still save you?

Check your answer

You do the real database lookup — the "maybe" could be a false positive, so the filter alone can't confirm a hit. What it bought you is every "definitely not": those queries skip the database entirely. For a mostly-miss workload (most keys don't exist), that's the bulk of the traffic — which is exactly when a Bloom filter is worth its bits.
