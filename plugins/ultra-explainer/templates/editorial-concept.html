Adding one cache server shouldn't move every key. Consistent hashing is the trick that makes it move almost none of them.

The naïve way to shard by key — hash(key) % N — has a brutal failure mode: change N and nearly every key lands somewhere new. For a cache, that's a stampede. Here's the idea that fixes it, and what it costs.

The mod-N trap

Shard a key with server = hash(key) % N and everything works — until N changes. Go from 4 servers to 5 and you haven't moved a fifth of the keys; you've moved roughly four-fifths of them, because the modulus rewrites almost every result at once.

what you'd expect

"Add a server, ~20% of keys relocate to it." Reasonable — and wrong.

what actually happens

k % 4 and k % 5 agree for only a handful of keys. The cache empties almost completely and every miss hits the origin at once.

Put the servers on a ring

Hash the servers onto a circle of hash values, not just the keys. Now both live in the same space: a server sits at hash(server), a key sits at hash(key), both points on one ring from 0 to 232.

A key belongs to the first server you meet walking clockwise from where the key landed. That single rule is the whole mechanism — no modulus, no global N.

Why a circle? So there's always a "next" server — walk clockwise past the top and you wrap to the lowest-valued server. No key ever falls off the end.
Reading a key
  1. Hash the key to a point on the ring — same hash function, same 0…232 space as the servers.
  2. Walk clockwise until you hit a server marker. In practice this is a binary search over a sorted list of server positions — O(log N), not O(N).
  3. That server owns the key. Every node runs the identical lookup, so they all agree without coordinating.
The payoff

Add a 5th server and only the keys in one arc move — the slice between the new server and its clockwise predecessor.

Every other key still meets the same server it always did. Instead of relocating ~80% of keys, you relocate on the order of 1/N of them. Removing a server is the mirror image: only its arc spills over to the next node clockwise. The cache stays warm.

What it costs

Consistent hashing isn't free, and the honest version names the catch:

  • With few servers, random placement makes arcs uneven — one node can own twice its share. The standard fix is virtual nodes: hash each server to many ring points so the load averages out.
  • It balances key count, not key heat. A single hot key still lands on one node; consistent hashing won't save you from that.
The one-line version. Hash servers and keys into the same ring; a key belongs to the next server clockwise. Membership changes then disturb one arc instead of the whole map — which is exactly the property a cache, a shard map, or a DHT needs.
